# 项目详述

## 一、项目概述与背景

### 1.1 项目背景

随着隐私计算技术的快速发展，秘密共享（Secret Sharing）作为安全多方计算的核心技术之一，在数据安全领域发挥着越来越重要的作用。秘密共享的核心思想是将一个秘密随机拆分为若干份额，只有持有足够份额数量的参与方聚集在一起才可以重建秘密。

本项目基于**Shamir门限秘密共享协议**，设计并实现了一套完整的**企业资金安全审批系统**。该系统将Shamir秘密共享应用于**审计条目密钥保护**，实现了：

- 审计条目的安全存储（无法恢复密钥则条目作废）
- 份额的动态更新（零值扰动，密钥保持不变）
- 灵活的份额数量（GF(2^32-5)域支持）
- 秘密原件在服务端与客户端之间的物理隔离

### 1.2 项目目标

本项目旨在开发一个基于Shamir秘密共享协议的通用隐私计算网关，实现：

1. **算法验证**：使用Python依次还原实数域、GF(257)、GF(2^32-5)域上的SSS算法标准模型和PSS特性
2. **套具开发**：开发支持PSS特性的GF(2^32-5)域算法套具（Python版、Go版、Dart版）
3. **系统实现**：基于GoFrame后端和Flutter前端实现完整的资金审批流程
4. **安全保证**：满足密码学系统的安全要求，确保敏感数据不泄露

### 1.3 项目特色

| 特色 | 描述 |
|------|------|
| **零值扰动历史管理** | 后端只保存零值扰动历史，不保存原始多项式或秘密 |
| **物理隔离设计** | 秘密原件在服务端与客户端之间物理隔离 |
| **GF(2^32-5)域** | 支持超过256份份额，适配AES密钥长度 |
| **前端本地运算** | 份额更新在前端本地完成，后端只转发扰动向量 |

## 二、核心算法原理

### 2.1 Shamir秘密共享原理

Shamir秘密共享方案是一种基于**多项式插值**的密码学算法，由Adi Shamir于1979年提出[1]。

#### 2.1.1 核心思想

```
多项式: f(x) = s + a₁x + a₂x² + ... + aₜ₋₁xᵗ⁻¹ (mod p)

其中:
- s: 秘密值（常数项）
- aᵢ: 随机系数
- t: 阈值（恢复秘密所需的最小份额数）
- p: 大素数（模数）
```

#### 2.1.2 秘密分发

1. 将秘密值s作为多项式的常数项
2. 生成一个t-1次多项式，其中t为阈值
3. 在不同点x=1,2,...,n处计算多项式值，得到n个份额
4. 将每个份额(x, f(x))分发给对应的参与方

#### 2.1.3 秘密重建

根据Lagrange插值公式，任意t个份额可以恢复原始秘密：

```
f(0) = Σᵢ f(xᵢ) · lᵢ(0) (mod p)

其中 lᵢ(0) 是Lagrange基多项式在0点的值
```

#### 2.1.4 为什么选择 Shamir 而非加性秘密共享？

| 特性 | 加性秘密共享 | Shamir秘密共享 |
|------|-------------|---------------|
| 门限支持 | 需要所有份额 | 支持任意t-of-n门限 |
| 份额大小 | 与秘密相同 | 与秘密相同 |
| 计算复杂度 | O(n) | O(t log²t) |
| 同态特性 | 天然支持加法 | 需要额外机制 |

**思考**：Shamir方案的优势在于支持任意门限设置，且份额可以预先计算。但其乘法运算需要Beaver三元组，复杂度较高。本项目主要使用加法同态特性，因此Shamir方案是理想选择。

### 2.2 有限域选择：GF(2^32-5)

#### 2.2.1 域选择理由

本项目选择模数为 **2³² - 5 = 4294967291**，原因如下：

| 特性 | 说明 |
|------|------|
| **最大32位安全素数** | 4294967291是小于2³²的最大安全素数 |
| **避免溢出** | 使用uint64进行中间运算，确保不溢出 |
| **高效计算** | 模数接近2³²，便于优化取模运算 |
| **安全性** | 作为安全素数，提供良好的密码学性质 |
| **份额扩展性** | 可支持超过256份份额（理论上限2³²-6） |

#### 2.2.2 与GF(257)的对比

| 特性 | GF(257) | GF(2^32-5) |
|------|---------|------------|
| 适用场景 | 图像像素（0-255） | 通用字节数组 |
| 份额数量限制 | ≤256 | 可超过2³² |
| 字节数组处理 | 需要复杂映射 | 直接分块处理 |
| Padding需求 | 可选 | PKCS#7风格 |

**架构决策**：由于本项目不再处理图像输入，而是保护AES密钥，因此选择GF(2^32-5)域。AES密钥通常为16/24/32字节，使用GF(2^32-5)可以直接处理，无需复杂的字节映射。

### 2.3 零值扰动多项式（Proactive Secret Sharing）

#### 2.3.1 问题引入

传统Shamir方案存在**被动泄露风险**：如果攻击者获得了某参与方的长期份额，随着时间推移，攻击者可能积累足够份额恢复秘密。

**解决思路**[2]：定期更新份额，但不改变秘密本身。

#### 2.3.2 零值扰动多项式原理

**定义**：零值扰动多项式g(x)是一个t-1次多项式，其常数项为0：

```
g(x) = 0 + b₁x + b₂x² + ... + bₜ₋₁xᵗ⁻¹ (mod p)
```

**关键特性**：g(0) = 0，因此添加到原始多项式后不会改变秘密值。

#### 2.3.3 份额更新过程

```
原始份额: f(i)
零值扰动: g(i)
新份额: f'(i) = f(i) + g(i) (mod p)

验证:
f'(0) = f(0) + g(0) = s + 0 = s ✓
```

#### 2.3.4 多次更新叠加

```
初始秘密多项式: f(x)，其中 f(0) = secret
第1次扰动: g₁(x)，其中 g₁(0) = 0
    → f₁(x) = f(x) + g₁(x)，f₁(0) = secret
第2次扰动: g₂(x)，其中 g₂(0) = 0
    → f₂(x) = f₁(x) + g₂(x) = f(x) + g₁(x) + g₂(x)，f₂(0) = secret
...
第n次扰动: gₙ(x)，其中 gₙ(0) = 0
    → fₙ(x) = f(x) + Σgᵢ(x)，fₙ(0) = secret
```

**数学保证**：无论进行多少次零值扰动，秘密值始终保持不变。

## 三、架构设计与决策思考

### 3.1 后端设计

#### 3.1.1 设计原则

后端架构遵循以下核心原则：

1. **不保存原始秘密**：后端不保存AES密钥、审计条目原件
2. **不保存用户份额**：后端不在任何时刻任何位置保存用户份额
3. **零值扰动历史存储**：只保存零值扰动历史，支持份额更新

#### 3.1.2 为什么后端不能保存原始份额？

**安全考量**：如果后端保存了用户份额，将面临以下风险：

| 风险类型 | 描述 | 后果 |
|---------|------|------|
| **内部威胁** | 运维人员或开发人员访问数据库 | 可以恢复原始秘密 |
| **外部攻击** | 数据库被SQL注入或拖库 | 攻击者获得所有秘密 |
| **内存泄露** | 后端缓存被读取 | 运行时秘密暴露 |

**密码学原则**：秘密共享的设计初衷是**物理隔离**——即使后端被完全攻破，攻击者也无法获得足够份额恢复秘密。如果后端保存了所有份额，这个安全属性就被破坏了。

**架构决策**：后端只负责协调和转发，不保存任何可用于恢复秘密的数据。

#### 3.1.3 为什么后端要保存随机数种子而不是原始多项式？

**直接保存原始多项式的问题**：

```
如果后端保存 f(x) = s + a₁x + ... + aₜ₋₁xᵗ⁻¹
则后端可以直接计算 f(0) = s
这相当于后端拥有了原始秘密！
```

**保存随机数种子的优势**：

| 方案 | 安全性 | 功能性 | 推荐度 |
|------|--------|--------|--------|
| 保存完整多项式 | ❌ 灾难性（等于保存秘密） | ✅ 完美 | ⛔ 禁止 |
| 保存随机数种子 | ⚠️ 中等（需加密存储） | ✅ 良好 | ⚠️ 需谨慎设计 |
| 不保存任何东西 | ✅ 最安全 | ❌ 无法更新 | 😢 不可行 |

**种子方案的安全性分析**：

1. **知道g(x)的seed无法直接得出秘密**——种子只是生成多项式的参数
2. **需要t个旧份额+g(x)才能生成新份额**——即使有种子，没有份额也无法工作
3. **后端只知道"如何更新"，不知道"原来是什么"**——数学上保证了安全性

#### 3.1.4 后端存储表结构

```
┌──────────────────────────────────────────────────┐
│ 零值扰动历史表 (zero_poly_history)              │
├──────────────────────────────────────────────────┤
│ id           │ 主键自增                         │
│ version      │ 扰动版本号（1, 2, 3...）         │
│ seed         │ 生成g(x)的种子（加密存储）        │
│ threshold    │ 门限值                           │
│ created_at   │ 创建时间戳                       │
│ delta_hash   │ 扰动向量的哈希摘要               │
└──────────────────────────────────────────────────┘
```

#### 3.1.5 为什么后端是单机加载份额计数器，而不是分布式同步？

**架构决策的背景**：

| 方案 | 复杂度 | 一致性保证 | 实现成本 |
|------|--------|-----------|----------|
| 分布式锁（Redis/ZooKeeper） | 高 | 强一致 | 需要额外基础设施 |
| 数据库行锁 | 中 | 强一致 | 简单 |
| 单机内存计数器 | 低 | 最终一致 | 几乎为零 |

**关键洞察**：份额提交流程本质上是**多阶段确认协议**，而不是高并发的数据写入。

**为什么不需要分布式同步？**

1. **场景特点**：份额提交是低频操作（用户主动触发），不是高并发写入
2. **一致性要求**：只需要保证"在5分钟内凑够t人"，而不是毫秒级一致性
3. **错误容忍**：超时未凑够可以简单重试，用户体验影响可接受

**架构决策**：使用**Fan-Out结构**在单个服务节点的不同协程间同步份额计数器，简化实现，降低复杂度。

```
┌─────────────────────────────────────────────────────────┐
│                    份额收集状态机                        │
├─────────────────────────────────────────────────────────┤
│                                                         │
│   用户A ──┐                                             │
│   用户B ──┼──→ [协程1] ──→ [全局计数器] ──→ [SSE通知]   │
│   用户C ──┘         ↑                    ↓             │
│                     └────── 超时控制 ◄───┘             │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

**注意**：如果未来需要多节点部署，可以引入Redis分布式锁，但当前单节点设计已经满足需求。

### 3.2 前端设计

#### 3.2.1 本地Vault系统

前端负责管理用户的份额文件，核心要求：

1. **原子写入**：保证Read-Update-Write的原子性
2. **事务支持**：利用SQLite事务实现崩溃回滚
3. **加密存储**：本地份额文件使用AES-256-GCM加密

#### 3.2.2 份额更新流程

```
后端下发: Δ（扰动向量，带版本号）
前端执行: f'(i) = f(i) + Δ[i] (mod 2^32-5)
```

**关键优势**：份额更新完全在本地完成，不涉及任何敏感数据在网络中的传输。

#### 3.2.3 本地加密文件结构

```
┌─────────────────────────────────────────────┐
│ Magic (4 bytes)    │ "SHAM"                │
├─────────────────────────────────────────────┤
│ Version (8 bytes)  │ 大端序版本号           │
├─────────────────────────────────────────────┤
│ UUID (16 bytes)    │ 份额唯一标识符        │
├─────────────────────────────────────────────┤
│ Length (4 bytes)   │ 份额数据长度          │
├─────────────────────────────────────────────┤
│ IV (12 bytes)      │ AES-GCM 初始化向量    │
├─────────────────────────────────────────────┤
│ Tag (16 bytes)     │ 认证标签              │
├─────────────────────────────────────────────┤
│ Data (N bytes)     │ 加密的份额数据        │
└─────────────────────────────────────────────┘
```

### 3.3 数据流程

#### 3.3.1 系统初始化流程

```
1. 生成 AES 密钥（16/24/32字节，4字节倍数，无需Padding）
2. 使用 GF(2^32-5) 域的 Shamir 算法分割密钥
3. 将份额分发到各用户（后端不存储任何多项式）
4. 前端保存份额到本地 Vault
5. 后端记录初始化完成状态
```

#### 3.3.2 审计条目加密流程

```
1. 产生审计条目（操作日志）
2. 使用 AES 密钥加密审计条目
3. 保存加密后的审计条目到数据库
4. 不保存审计条目原件
5. 记录加密操作日志
```

**关键设计**：后端保存的是加密后的审计条目，无法在没有密钥的情况下读取内容。

#### 3.3.3 份额提交与恢复流程

```
1. 前端点击按钮，通知后端进入"等待份额全部提交"状态
2. 后端启动状态机协程循环，设置5分钟超时
3. 后端更新份额状态：已有n人准备好提交，还需x人提交
4. 其他用户加入时，Fan-Out模块同步更新全局计数器
5. 凑够份额数后，后端通过SSE发送通知
6. 后端接收到足够份额后，执行拉格朗日插值恢复AES密钥
7. 使用密钥解密加密的审计条目
8. 审计条目对所有人开放
9. 前端内存验证后立即擦除密钥和审计条目
```

## 四、安全性分析

### 4.1 零值扰动历史存储原则

| 原则 | 描述 |
|------|------|
| **不保存原始多项式** | 后端只保存零值扰动历史，不保存原始秘密 |
| **种子加密存储** | 每个零值扰动的种子单独加密，防止未授权访问 |
| **历史可追溯** | 重启后可从历史表恢复扰动序列 |

### 4.2 不保存秘密原则

| 要求 | 实现 |
|------|------|
| 不保存原始密钥 | AES密钥生成后立即分发给用户，后端不保留 |
| 不保存审计原件 | 审计条目使用AES加密后存储，解密后不在后端留存 |
| 不保存用户份额 | 份额只在用户本地存储，后端只在内存中暂存用于恢复 |

### 4.3 隐私保护机制

| 机制 | 描述 |
|------|------|
| **端到端加密** | 后端无法解密份额内容 |
| **内存安全** | AES密钥恢复只在前端内存中完成 |
| **自动清理** | 30秒无操作自动擦除内存 |
| **零持久化** | 审计条目解密后只在内存中展示，不落盘 |

### 4.4 密码学安全保障

**即使数据库泄露，攻击者也无法**：

1. ❌ 从扰动历史中还原原始秘密——不知道原始多项式f(x)
2. ❌ 从种子直接得出秘密——种子只是生成参数
3. ❌ 伪造新份额——需要t个旧份额才能生成新份额

**攻击者需要同时获得**：
- t个旧份额（从前端泄露）
- 当前扰动历史（从后端数据库）
- 完整的拉格朗日插值计算能力
