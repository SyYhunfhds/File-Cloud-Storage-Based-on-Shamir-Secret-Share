# 基于Shamir算法的企业财务审计条目存储系统设计规划

## 摘要

本毕业设计项目旨在设计并实现一个基于Shamir门限秘密共享算法的企业财务审计条目安全存储系统。系统以Privy.io为参考架构，结合自主研发的GF(2^32-5)域Shamir算法实现和零值扰动多项式（Proactive Secret Sharing, PSS）特性，构建一套完整的前后端分离架构。项目后端采用GoFrame框架，前端采用Flutter桌面端技术，数据库采用PostgreSQL与SQLite混合架构。

本设计文档从架构统筹、算法设计、前端设计、后端设计四个维度进行详细阐述，分析现有代码基础、Privy.io参考架构的技术特点，并提出具体的设计方案和工作量估算。

**关键词**：Shamir秘密共享；零值扰动多项式；GF(2^32-5)有限域；GoFrame；Flutter；企业财务审计

## 第一章 项目概述与研究背景

### 1.1 项目背景

随着隐私计算技术的快速发展，秘密共享（Secret Sharing）作为安全多方计算的核心技术之一，在数据安全领域发挥着越来越重要的作用。秘密共享的核心思想是将一个秘密随机拆分为若干份额（shares），只有持有足够份额数量的参与方聚集在一起才可以重建秘密[1]。

在企业财务审计场景中，敏感审计条目的保护至关重要。传统的加密方案存在单点故障风险——一旦密钥泄露，所有审计条目将面临暴露风险。而基于Shamir秘密共享的方案可以将密钥分散管理，即使部分份额泄露，攻击者仍无法恢复原始秘密。

本项目基于**Shamir门限秘密共享协议**，设计并实现了一套完整的企业财务审计条目安全存储系统。该系统将Shamir秘密共享应用于**审计条目加密密钥保护**，实现了：

- 审计条目的安全存储（无法恢复密钥则条目作废）
- 份额的动态更新（零值扰动，密钥保持不变）
- 灵活的份额数量（GF(2^32-5)域支持）
- 秘密原件在服务端与客户端之间的物理隔离

### 1.2 项目目标

本项目旨在开发一个基于Shamir秘密共享协议的通用隐私计算网关，实现：

1. **算法验证**：使用Python依次还原实数域、GF(257)、GF(2^32-5)域上的SSS算法标准模型和PSS特性
2. **套具开发**：开发支持PSS特性的GF(2^32-5)域算法套具（Python版、Go版、Dart版）
3. **系统实现**：基于GoFrame后端和Flutter前端实现完整的资金审批流程
4. **安全保证**：满足密码学系统的安全要求，确保敏感数据不泄露

## 第二章 参考架构研究——Privy.io分析

### 2.1 Privy.io概述

Privy是一家专注于嵌入式钱包基础设施的企业级Web3钱包平台，截至2025年已服务超过1.2亿账户、2000+开发团队[2]。其核心技术架构被多家大型企业采用，包括支付巨头Stripe的收购。

Privy的核心价值在于提供**硬件隔离的自托管钱包**解决方案，将复杂的密钥管理抽象为简单的API调用，使开发者无需深入密码学底层即可实现安全的钱包功能。

### 2.2 Privy.io核心架构

#### 2.2.1 密钥生成分片机制

Privy的钱包基础设施基于**分布式密钥分片**（Distributed Key Sharding）设计。当用户创建钱包时，私钥在TEE（Trusted Execution Environment，可信执行环境）内生成，并立即被分割为两个份额：

1. **Enclave Share**：由TEE保护，只能在TEE内部访问
2. **Auth Share**：由Privy API存储，需要有效认证才能检索

两个份额形成**2-of-2门限集**，即两个份额同时存在才能重建密钥。单独一个份额既不能透露密钥信息，也不能获取钱包访问权限[3]。

```
┌─────────────────────────────────────────────────────────────┐
│                    钱包创建流程                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   用户认证 ──→ TEE生成私钥 ──→ 分割为2个份额              │
│                              ├── Enclave Share (TEE保护)    │
│                              └── Auth Share (API存储)       │
│                                                             │
│   两者形成2-of-2门限，必须同时存在才能重建密钥              │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```
#### 2.2.2 安全执行环境

Privy的钱包基础设施设计使得**完整的私钥只存在于安全 enclave 内部**[3]。系统依赖硬化的可信执行环境来封装最敏感的操作：密钥生成、策略执行和签名。

TEE的核心安全特性：

| 特性 | 描述 |
|------|------|
| **硬件隔离** | CPU级沙箱，无持久化存储，无交互访问，内存加密 |
| **防御深度** | 每个钱包使用Shamir秘密共享生成加密分片，配合授权签名和策略检查 |
| **度量启动** | 每个enclave从签名镜像启动，验证证明报告后才初始化密钥 |

#### 2.2.3 请求处理流程

Privy的请求处理流程如下[3]：

```
1. 客户端调用Privy API
         ↓
2. API（在enclave外运行）检索对应分片的加密份额
         ↓
3. 只有enclave能将传入的分片与其自己的加密分片组合
         ↓
4. 在enclave内部验证授权签名和策略
         ↓
5. 所有检查通过后，enclave生成签名并立即丢弃重组的密钥
```

由于API和enclave位于不同的信任边界，单独攻破一个提供商不足以访问私钥。

#### 2.2.4 认证与会话管理

Privy采用多层认证机制：

- **访问令牌（Access Token）**：JWT格式，由Privy Ed25519密钥签名，有效期1小时
- **刷新令牌（Refresh Token）**：有效期30天，只能使用一次
- **认证方法**：邮箱/手机号OTP、社交登录（Google、Apple等）、钱包签名（SIWE）

### 2.3 Privy.io安全特性分析

#### 2.3.1 防御深度策略

Privy实施防御深度策略，避免单点故障：

| 安全层级 | 技术手段 |
|----------|----------|
| 密钥分片 | Shamir秘密共享，份额分布在不同基础设施 |
| 硬件隔离 | TEE保护enclave分片 |
| 传输加密 | TLS 1.2+最小支持 |
| 访问控制 | RBAC + 微分段，最小权限原则 |
| 数据库加密 | 静态加密 + 每日快照 + 5分钟增量日志 |

#### 2.3.2 密钥永不持久化

Privy的核心设计原则：**完整密钥从不存储或持久化**[2]。

- 密钥只在签名时短暂重组于enclave内存中
- 签名完成后立即擦除
- 既不存储于数据库，也不缓存于磁盘

#### 2.3.3 策略引擎

Privy提供可编程的策略控制：

- 签名策略：要求生物识别、限制设备、定义交易限额
- 恢复定制：备用方法、会话过期、多设备支持
- 审批工作流：交易确认、委托访问

## 第三章 算法设计与实现

### 3.1 Shamir秘密共享原理

#### 3.1.1 算法概述

Shamir秘密共享方案是一种基于**多项式插值**的密码学算法，由Adi Shamir于1979年提出[1]。其核心思想是将秘密s作为t-1次多项式的常数项：

```
f(x) = s + a₁x + a₂x² + ... + aₜ₋₁xᵗ⁻¹ (mod p)
```

其中：
- s：秘密值（常数项）
- aᵢ：随机系数
- t：阈值（恢复秘密所需的最小份额数）
- p：大素数（模数）

#### 3.1.2 秘密分发与恢复

| 阶段 | 过程 |
|------|------|
| 分发 | 在x=1,2,...,n处计算多项式值，得到n个份额(x, f(x)) |
| 恢复 | 根据Lagrange插值公式，任意t个份额可以恢复原始秘密 |

**Lagrange插值公式**：

```
f(0) = Σᵢ f(xᵢ) · lᵢ(0) (mod p)

其中 lᵢ(0) = Πⱼ≠ᵢ xⱼ/(xⱼ - xᵢ) (mod p)
```

### 3.2 有限域选择：GF(2^32-5)

#### 3.2.1 域参数

本项目选择模数为 **2³² - 5 = 4294967291**，具有以下特性：

| 特性 | 说明 |
|------|------|
| **最大32位安全素数** | 4294967291是小于2³²的最大安全素数 |
| **避免溢出** | 使用uint64进行中间运算，确保不溢出 |
| **高效计算** | 模数接近2³²，便于优化取模运算 |
| **安全性** | 作为安全素数，提供良好的密码学性质 |
| **份额扩展性** | 可支持超过256份份额（理论上限2³²-6） |

#### 3.2.2 域运算实现

有限域GF(2^32-5)的基本运算实现如下：

```go
// 模加：使用分支避免溢出
func add(a, b uint32) uint32 {
    sum := uint64(a) + uint64(b)
    if sum >= prime {
        sum -= prime
    }
    return uint32(sum)
}

// 模减：使用分支避免下溢
func sub(a, b uint32) uint32 {
    if a >= b {
        return a - b
    }
    return prime - (b - a)
}

// 模乘：利用uint64避免溢出
func mul(a, b uint32) uint32 {
    product := uint64(a) * uint64(b)
    return uint32(product % uint64(prime))
}

// 模逆：基于费马小定理 a^(p-2) ≡ a^(-1) (mod p)
func inv(a uint32) uint32 {
    return pow(a, prime-2)
}
```

### 3.3 零值扰动多项式（Proactive Secret Sharing）

#### 3.3.1 问题引入

传统Shamir方案存在**被动泄露风险**：如果攻击者获得了某参与方的长期份额，随着时间推移，攻击者可能积累足够份额恢复秘密。

**解决思路**[4]：定期更新份额，但不改变秘密本身。

#### 3.3.2 零值扰动原理

零值扰动多项式g(x)是一个t-1次多项式，其常数项为0：

```
g(x) = 0 + b₁x + b₂x² + ... + bₜ₋₁xᵗ⁻¹ (mod p)
```

**关键特性**：g(0) = 0，因此添加到原始多项式后不会改变秘密值。

```
原始份额: f(i)
零值扰动: g(i)
新份额:   f'(i) = f(i) + g(i) (mod p)

验证:
f'(0) = f(0) + g(0) = s + 0 = s ✓
```

#### 3.3.3 多次更新叠加

```
初始秘密多项式: f(x)，其中 f(0) = secret
第1次扰动: g₁(x)，其中 g₁(0) = 0
    → f₁(x) = f(x) + g₁(x)，f₁(0) = secret
第2次扰动: g₂(x)，其中 g₂(0) = 0
    → f₂(x) = f₁(x) + g₂(x) = f(x) + g₁(x) + g₂(x)，f₂(0) = secret
...
第n次扰动: gₙ(x)，其中 gₙ(0) = 0
    → fₙ(x) = f(x) + Σgᵢ(x)，fₙ(0) = secret
```

#### 5.1.3 扰动生成模块

**Delta生成算法**：

```go
func GenerateDeltaVector(seed uint64, threshold int, userXs []uint32) DeltaVector {
    // 1. 使用HMAC-SHA256从种子生成随机系数
    coeffs := generateRandomCoefficients(seed, threshold-1)
    coeffs = append([]uint32{0}, coeffs...) // 常数项为0

    // 2. 对每个用户计算 g(userX)
    delta := make(DeltaVector, len(userXs))
    for i, x := range userXs {
        delta[i] = evalPolynomial(coeffs, x)
    }
    return delta
}
```

## 第六章 安全性设计

### 6.1 密码学安全保障

#### 6.1.1 零值扰动安全性

**即使数据库泄露，攻击者也无法**：
- ❌ 从扰动历史中还原原始秘密——不知道原始多项式f(x)
- ❌ 从种子直接得出秘密——种子只是生成参数
- ❌ 伪造新份额——需要t个旧份额才能生成新份额

**攻击者需要同时获得**：
- t个旧份额（从前端泄露）
- 当前扰动历史（从后端数据库）
- 完整的拉格朗日插值计算能力

#### 6.1.2 物理隔离原则

| 原则 | 实现 |
|------|------|
| 后端不保存秘密 | AES密钥生成后立即分发给用户 |
| 后端不保存份额 | 份额只在用户本地存储 |
| 后端不保存原件 | 审计条目加密后存储 |

### 6.2 传输安全

| 措施 | 说明 |
|------|------|
| TLS 1.2+ | 所有传输加密 |
| HTTPS | 生产环境强制 |
| JWT签名 | 防篡改认证令牌 |

### 6.3 存储安全

| 层级 | 技术 |
|------|------|
| 后端数据库 | PostgreSQL静态加密 |
| 本地份额文件 | AES-256-GCM加密 |
| 密钥派生 | Argon2id |
