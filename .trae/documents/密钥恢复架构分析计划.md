# 密钥恢复软件架构分析计划

## 一、任务概述

本计划旨在完成以下任务：
1. 调研开源环境/生产环境中密钥恢复软件的设计架构
2. 分析中心化保存多项式和去中心化MPC的优缺点及项目占比
3. 详细记录当前项目遇到的架构难题
4. 讨论架构可解释性与密码学全盲要求的权衡

## 二、调研计划

### 2.1 开源密钥恢复软件调研

#### 调研对象
- **HashiCorp Vault**：企业级密钥管理系统
- **ReMemory**：基于Shamir的数字遗产工具（已在搜索结果中发现）
- **libgfshare**：成熟的Shamir秘密共享库
- **Shard GUI**：基于Shamir的文件加密工具
- **KeePassXC**：开源密码管理器（可能使用秘密共享）
- **Bitwarden**：开源密码管理服务

#### 调研内容
1. 架构设计模式
2. 秘密存储策略
3. 份额管理方案
4. 安全保障措施
5. 开源社区活跃度

### 2.2 中心化 vs 去中心化架构分析

#### 维度
1. 安全性
2. 可用性
3. 可扩展性
4. 实现复杂度
5. 维护成本
6. 审计透明度

#### 对比框架
```
| 维度 | 中心化（保存多项式/种子） | 去中心化（MPC/全盲） |
|------|------------------------|------------------|
| 安全性 | ⚠️ 单节点风险 | ✅ 分布式信任 |
| 可用性 | ✅ 高可用 | ⚠️ 节点协调复杂 |
| 实现难度 | ✅ 简单 | ⚠️ 复杂 |
| 可审计性 | ✅ 清晰 | ⚠️ 抽象 |
| 性能 | ✅ 高效 | ⚠️ 通信开销大 |
```

### 2.3 项目占比分析

通过以下渠道获取数据：
1. GitHub Stars/Contributors统计
2. npm/pip包下载量
3. 学术论文引用

## 三、架构难题文档内容规划

### 3.1 文档结构

```
架构难题分析文档/
├── 1. 问题背景.md
├── 2. 关键抉择.md
├── 3. 权衡分析.md
├── 4. 推荐方案.md
└── 5. 实施路线图.md
```

### 3.2 详细内容

#### 问题背景
- 当前架构的设计目标
- 已识别的安全风险
- 业务需求约束

#### 关键抉择（核心难题）

##### 抉择1：后端是否保存多项式？
- **保存的理由**
  - 可解释性强，架构清晰
  - 便于调试和故障排查
  - 支持复杂的份额更新策略
- **不保存的理由**
  - 零知识原则的数学保证
  - 即使后端被攻破，秘密仍然安全
  - 符合密码学最佳实践

##### 抉择2：后端是否保存随机数种子？
- **保存的理由**
  - 支持份额的可追溯更新
  - 便于实现版本控制和回滚
  - 满足业务审计需求
- **不保存的理由**
  - 减少攻击面
  - 降低数据泄露风险
  - 极致的密码学全盲

##### 抉择3：后端是否暂存份额？
- **暂存的理由**
  - 提高可用性和用户体验
  - 支持复杂的恢复流程
  - 便于实现份额验证
- **不暂存的理由**
  - 彻底的物理隔离
  - 零信任原则
  - 防止内存泄露

#### 抉择4：可解释性 vs 全盲性
- **可解释性的价值**
  - 降低开发和维护成本
  - 便于代码审查和安全审计
  - 易于向非技术人员解释
- **全盲性的价值**
  - 最强的数学安全保证
  - 符合学术密码学标准
  - 适用于高安全等级场景
