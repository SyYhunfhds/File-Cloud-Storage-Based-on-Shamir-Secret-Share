# Rule: State & Data Persistence Protocol

你必须在涉及以下场景时停止自动生成代码，并向用户确认具体流程：

### 1. 表单管理流 (Form Management Flow)
当检测到页面包含多个输入组件（TextField, Checkbox, Dropdown）时，**必须询问**：
*   **校验时机**：是实时校验（OnChange）、失去焦点校验（OnBlur）、还是点击提交时统一校验？
*   **提交行为**：提交后是保持现有数据、重置表单、还是跳转页面？
*   **意外中断处理**：如果用户未保存就切换页面或关闭窗口，是否需要弹出拦截警告（WillPopScope/WindowListener）？

### 2. 持久化策略 (Local Persistence)
当功能涉及“设置”、“用户信息”或“离线数据流”时，**禁止直接选定方案**。请给出建议并询问用户：
*   **数据规模**：
    *   如果是配置信息（Key-Value），建议使用 `shared_preferences`。
    *   如果是结构化业务数据（List/Relational），建议使用 `sqlite (sqflite)` 或 `isar`。
    *   如果是超大文件或日志，建议直接使用 `path_provider` 操作本地文件。
*   **同步频率**：是“修改即保存（Auto-save）”，还是“用户点击保存按钮后批量写入”？

### 3. 跨页面同步 (Global State)
当数据需要被多个大屏组件共享时，**必须询问**：
*   **生命周期**：数据是仅在当前模块存活，还是全局单例（如登录状态、系统全局设置）？
*   **响应式需求**：UI 是否需要随底层数据变更立即刷新？（决定使用 `Stream` 还是普通的 `Notifier`）。

---

### Agent 提问话术示例 (Stop & Ask Templates)

*   *"检测到该页面是一个复杂表单，请问：我应该在用户输入时即时验证错误，还是等用户点击‘保存’后再验证？"*
*   *"该页面的数据需要持久化到本地吗？如果需要，您倾向于使用 SQLite 数据库还是简单的 JSON 配置文件？"*
*   *"当用户在大屏上修改了左侧列表数据，右侧详情页是否需要实时同步更新？"*

---

**执行准则：**
1.  **先沟通流程，再输出 StateNotifier/Provider 代码。**
2.  **严禁在大屏软件中默认使用“内存存储”来处理关键业务数据，必须确认持久化需求。**
