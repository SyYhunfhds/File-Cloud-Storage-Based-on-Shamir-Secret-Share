# 附录 && 补充要求
1. 参考书写风格: [书写风格](frontend\.trae\documents\书写风格.md)
2. 项目参考`privy.io`的 Shamir 2-of-3 架构, 但是进行了很多简化, 并且根据项目情况进行了*安全降级*
    - 移除了对SSS份额的AES加密处理, 原因参考[后端 BUG汇总文档](backend\issues.md); 当前计划使用XXTEA对份额坐标进行**保格式**混淆, 项目里面没有任何代码能体现这个规划, 你就放到*展望*里就可以了
3. 项目结构大致如下:
    - `backend`: GoFrame后端
        - `pkg/shamir`: 提供Shamir uint32域的两个实现, `v2`为完整版, 需要全程传入随机数种子; `v3`为简化版, 不需要传入随机数种子
    - `frontend`: Flutter桌面端实现
        - 由于使用的skill的问题, 项目里有时候用的是json serializable插件, 有时候用的是switch case模式匹配; 可将其归入 项目不足->可维护性 && Ai辅助系统开发 中
    - `exploration`: 根据子目录创建时间, 分别为Shamir算法实数域的Python验证、Shamir 伽罗瓦域实现、Shamir GF(257)域实现、Shamir 图像特化输入验证实现、Shamir算法的伽罗瓦域Go实现

    其次, `backend/.trae/documents`和`frontend/.trae/documents`目录下的文档主要为Ai开发和修复前端的规划文档, 可以用于项目工作量佐证
4. Insomnia API测试文档不包含在项目中, 忽略API单元测试的内容
5. **具体实现**这一节中必须从项目中找出源代码
6. 后端API大部分都有链路追踪, 个别API还有敏感Span——可以归为 `软件测试-链路追踪` 和 `项目不足-没有移除的调试语句`, 这是个**双刃剑**
